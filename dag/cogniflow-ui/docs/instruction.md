# Cogniflow UI — Internal Instruction Manual

This document explains, in detail:

1. How the bundled-pipeline **seeding** works on every startup.
2. How each **configuration file** maps to a runtime behavior.
3. How to add or update bundled pipelines and ship a new release.

It is the source of truth for anyone changing how the UI initializes the
orchestrator's pipelines folder. If you only want a quick orientation, read
[`../readme.md`](../readme.md) first.

---

## 1. Seeding — the overlay model

### 1.1 Why the UI seeds anything at all

`dag/cogniflow-ui` is intended to ship as a `.exe` / `.app` for non-technical
students. They install the UI executable, install the orchestrator executable
separately, and expect to immediately see working examples — not an empty
"Pipelines" page.

The UI therefore carries its own copy of curated example pipelines in
`seed_pipelines/` and overlays them onto the orchestrator's `pipelines/`
folder when the FastAPI app boots. The result: on first launch the user
sees the bundled examples; on subsequent launches their edits stick.

### 1.2 What `seed_pipelines/` contains

`seed_pipelines/<name>/` is a **trimmed copy** of an authored pipeline. It
includes only the files that define the pipeline, and excludes everything
that the orchestrator generates at runtime.

Included in seeds:

| Path | Purpose |
|---|---|
| `pipeline.json` | Pipeline definition (agents, edges, settings). |
| `agents/<id>/01_system.md` | System prompt for the agent. |
| `agents/<id>/02_prompt.md` | Task prompt for the agent. |
| `agents/<id>/03_inputs/` | Optional input files used by the agent (full subtree). |

Excluded from seeds (these are generated by the orchestrator and would be
stale or destructive if overwritten):

| Path | Why excluded |
|---|---|
| `agents/<id>/04_context.md` | Built at run time from inputs and predecessors. |
| `agents/<id>/05_output.md`, `05_output.<ts>.md`, `05_output.pre_v*.md` | Run output and rotated copies. |
| `agents/<id>/05_usage.json`, `05_usage.<ts>.json` | Token-usage telemetry. |
| `agents/<id>/06_status.json` | Live agent status. |
| `events.jsonl`, `summary.json`, `env.snapshot.json`, `command`, `pause`, `resume` | Runtime sentinels and event stream. |
| `history/<run_id>/` | Per-run snapshots produced by the observer. |
| `.state/`, `.configurator/` | Internal state directories. |

The trim is performed once, by hand, when curating a new bundled pipeline
(see §3.2 for the exact procedure).

### 1.3 The seed marker

Inside the live `pipelines_root` (the orchestrator's pipelines folder), the
UI maintains a single bookkeeping file:

```
<pipelines_root>/.ui-seed-marker.json
```

```json
{
  "app_version": "1.0.0",
  "last_seeded_at": "2026-04-25T16:27:59.181089+00:00",
  "seeded": [
    "01-writer-critic",
    "04-code-review-board",
    "07-newspaper-ai-article",
    "10-software-factory",
    "15-universe-origins"
  ]
}
```

The UI never writes any other state into the pipelines folder. The marker is
small, human-readable, and safe to delete (a deletion forces the next launch
to re-seed everything as if it were the first launch).

### 1.4 Lifecycle — what happens on every startup

When the FastAPI lifespan hook runs, `seeding.seed_pipelines()` is called
with three inputs: `seed_root` (this package's `seed_pipelines/`),
`pipelines_root` (resolved from the observer config), and the current
`app_version` (from the top-level `config.json`).

It then chooses one of four branches:

| Marker state at launch | UI's `app_version` | Result |
|---|---|---|
| Marker file is missing | any | **Seed everything.** Copy every `seed_pipelines/<name>/` into `<pipelines_root>/<name>/`. Write the marker. |
| Marker present, version equal | unchanged | **Skip.** Pipeline files are not touched. The boot log prints `[seed] skipped (...)`. |
| Marker present, version different | bumped (e.g. 1.0.0 → 1.1.0) | **Re-seed.** Copy every `seed_pipelines/<name>/` again, overwriting same-named files. Update the marker. |
| Marker present, but `seed_pipelines/` missing on disk (broken bundle) | any | **Skip.** Marker is not modified, so the next launch will retry. |

Two further invariants hold across all branches:

* **User-created or lab-added pipelines are never touched.** The seeder
  only writes paths underneath `<pipelines_root>/<seed_name>/` where
  `<seed_name>` is a directory currently present in `seed_pipelines/`. A
  pipeline named `99-my-lab-pipeline` that exists in `pipelines_root` but
  not in `seed_pipelines/` is invisible to the seeder.
* **Runtime artefacts inside seeded pipelines are preserved.** The seeder
  performs an additive copy — it writes the files that exist in the bundle
  but never deletes files that the orchestrator wrote. So an existing
  `01-writer-critic/agents/001_writer/05_output.md` survives a re-seed
  even when its sibling `01_system.md` is overwritten.

### 1.5 Worked example — what students experience

| Event | What's on disk | What the UI does next launch |
|---|---|---|
| Student installs UI v1.0.0 against an empty `pipelines/` | only `.ui-seed-marker.json` is missing | First-run seed: 5 bundled pipelines copied; marker written with `app_version: 1.0.0`. |
| Student edits the writer's `01_system.md` and restarts | marker present, same version | Skip. Their edit persists. |
| Student creates `99-my-lab-pipeline` via the UI and restarts | extra pipeline added by user | Skip. Their pipeline is untouched. |
| You ship UI v1.1.0 with an updated writer prompt | `app_version` in `config.json` is bumped | Re-seed. The student's local edit to `01_system.md` is overwritten by the new bundled version, but `99-my-lab-pipeline` and the previously generated `05_output.md` are preserved. |
| Student manually deletes the marker | marker is gone | Treated as first-run. All five bundled pipelines are refreshed; user pipelines untouched. |

### 1.6 What the seeder will *not* do

* It will not delete files. There is no "remove a bundled pipeline" path
  during seeding. If you decide to drop `15-universe-origins` from a
  future release, the previously seeded copy stays on disk for any student
  who already installed v1.0.0 — which is correct behavior, since their
  edits and run history shouldn't disappear because of a curation change.
* It will not migrate or rename pipelines. If you need to rename a bundled
  pipeline, ship both names side-by-side for one release and document the
  switch.
* It does not touch `pipelines_root/templates/` (the configurator's
  template store). Templates have their own management UI under
  `/templates`.

## 2. Configuration — what each file controls

The UI reads three `config.json` files. They are layered: top-level → observer
→ configurator. Each governs a specific subset of the runtime, and their
fields do not overlap.

### 2.1 Top-level — `dag/cogniflow-ui/config.json`

This is the file the **student** edits. In a packaged build it lives next to
the `.exe` / `.app` so they can change the port or the version pointer
without unpacking the bundle.

```json
{
  "app_title": "Cogniflow",
  "app_version": "1.0.0",
  "host": "127.0.0.1",
  "port": 8000
}
```

| Field | Default | Effect |
|---|---|---|
| `app_title` | `"Cogniflow"` | Window title used by FastAPI / browser tab. |
| `app_version` | `"0.0.0"` | **Drives the seeding decision.** Compared against the marker file in `pipelines_root` to decide between skip, first-run seed, or re-seed. Bump this whenever you ship updated bundled pipelines. |
| `host` | `"127.0.0.1"` | Bind address. Keep on loopback so other machines on the same Wi-Fi cannot reach the student's webapp. |
| `port` | `8000` | TCP port. Change if 8000 is taken (rare on a clean student machine). |

### 2.2 Observer — `dag/cogniflow-ui/observer/config.json`

```json
{
  "orchestrator_root": "../../dag/cogniflow-orchestrator",
  "poll_interval_ms": 2000,
  "model_context_limit": 180000,
  "event_tail_lines": 50,
  "app_title": "Cogniflow Observer",
  "approver": "operator",
  "host": "127.0.0.1",
  "port": 8000,
  "versioning": true
}
```

| Field | Effect |
|---|---|
| `orchestrator_root` | **Source of truth for `pipelines_root`.** The observer computes `pipelines_root = orchestrator_root / "pipelines"`. The seeder reads this same value. Relative paths resolve against this `config.json`'s directory; absolute paths are used as-is. In a packaged build, point this at wherever the student installed the orchestrator. |
| `poll_interval_ms` | How often the board polls for status updates (HTMX `hx-trigger="every Xms"`). |
| `model_context_limit` | Token-budget warning threshold for the cards view. |
| `event_tail_lines` | How many tail lines of `events.jsonl` the events panel shows. |
| `app_title` | Title shown on observer pages. (Not the same field as the top-level title; FastAPI uses the top-level one.) |
| `approver` | Logged in approval files when a user clicks Approve / Reject. |
| `host` / `port` | **Ignored** in the merged build. The unified app uses the top-level `config.json` values for binding. Left for forward compatibility with running the observer standalone. |
| `versioning` | Gates the History feature. When `false`, all `/pipelines/*/history*` routes return 404. |

### 2.3 Configurator — `dag/cogniflow-ui/configurator/config.json`

```json
{
  "pipelines_root": "../../dag/cogniflow-orchestrator/pipelines",
  "cfg_port": 8001,
  "cfg_max_versions": 50,
  "cfg_readonly": false,
  "model_context_limit": 180000,
  "cfg_trash_dir": "",
  "app_title": "Cogniflow Configurator",
  "prompt_templates_dir": "",
  "claude_bin": "",
  "enable_specialize": true,
  "meta_prompt_timeout_s": 300,
  "default_taglines_system": ["role", "responsibilities", "guardrails"],
  "default_taglines_task":   ["description", "goals", "context", "input", "output", "format", "guardrails"]
}
```

| Field | Effect |
|---|---|
| `pipelines_root` | **Configurator's view of the pipelines folder.** Should resolve to the same directory as `orchestrator_root + "/pipelines"` from the observer config. Relative paths resolve against this `config.json`'s directory. The configurator and the observer must agree, otherwise edits in one are invisible to the other. |
| `cfg_port` | **Ignored** in the merged build. Kept for forward compatibility. |
| `cfg_max_versions` | Max retained versions per file in `<pipeline>/.configurator/history/`. Older snapshots are pruned beyond this. |
| `cfg_readonly` | When `true`, all write endpoints are disabled and the UI hides edit affordances. |
| `model_context_limit` | Token estimate target shown in the prompt editor. |
| `cfg_trash_dir` | Where deleted pipelines move. Empty value → `<pipelines_root>/.trash/`. |
| `app_title` | Title shown on configurator pages. |
| `prompt_templates_dir` | Override the default `prompt_templates/` location. Empty → use the bundled package directory (`configurator/prompt_templates/`). |
| `claude_bin` | Explicit path to the `claude` CLI for the meta-prompt Specialize feature. Empty → search PATH and platform conventions. When not found, the UI hides the Specialize button gracefully. |
| `enable_specialize` | Master switch for the Specialize / "Generate" feature. `true` (default) → button visible when `claude` CLI is found. `false` → button is never shown anywhere in the UI, regardless of CLI availability. Use this to disable the feature in environments where the CLI is installed but the feature is not desired. |
| `meta_prompt_timeout_s` | Hard timeout for the Specialize subprocess. |
| `default_taglines_system` | Globally required taglines for `01_system.md` files (override per-pipeline in `pipeline.json`). |
| `default_taglines_task` | Globally required taglines for `02_prompt.md` files. |

Every field except `pipelines_root`, `app_title`, and the tagline lists also
accepts an environment-variable override (uppercase form) — useful for
testing without editing the file.

### 2.4 Path resolution rules

The observer and configurator each resolve their pipelines path relative to
the directory holding their *own* `config.json`. Concrete examples:

| Setting | `config.json` lives at | Resolves to |
|---|---|---|
| `"orchestrator_root": "../../dag/cogniflow-orchestrator"` | `dag/cogniflow-ui/observer/config.json` | `dag/cogniflow-orchestrator/` (one level up from `dag/cogniflow-ui/`) |
| `"pipelines_root": "../../dag/cogniflow-orchestrator/pipelines"` | `dag/cogniflow-ui/configurator/config.json` | `dag/cogniflow-orchestrator/pipelines/` |
| `"pipelines_root": "C:/data/pipelines"` | anywhere | `C:/data/pipelines/` (absolute paths bypass the relative rule) |

For the packaged build, both fields can point at the orchestrator's installed
pipelines directory, e.g. `~/Cogniflow/orchestrator/pipelines`.

### 2.5 How configuration relates to seeding

The seeder needs three pieces of information at startup. Each comes from a
specific config file:

| Seeder input | Source | If wrong |
|---|---|---|
| `app_version` | top-level `config.json` | Same-version restarts re-seed (annoying but harmless), or version-bumps fail to refresh (silent stale prompts). |
| `pipelines_root` | observer `config.json` → `orchestrator_root/pipelines` | Seeds land in the wrong folder; observer / configurator look elsewhere; user sees an empty home page. |
| `seed_pipelines/` directory | physical layout next to `app.py` | Missing dir → seeder skips silently, leaves marker untouched, retries next launch. |

If you run the UI and see no example pipelines on the home page, check those
three inputs in order: top-level version, observer's `orchestrator_root`,
and the actual presence of `seed_pipelines/`.

## 3. Operational recipes

### 3.1 Force a re-seed without bumping the version

Delete the marker:

```
rm <pipelines_root>/.ui-seed-marker.json
```

The next launch is treated as a first-run seed. This is the standard way to
recover from a corrupted bundled pipeline on a single student's machine.

### 3.2 Add or update a bundled pipeline

1. Author the pipeline in the live UI (or copy from `dag/cogniflow-orchestrator/pipelines/`).
2. Strip runtime artefacts to produce a clean seed copy. Keep only:
   * `pipeline.json`
   * `agents/<id>/01_system.md`
   * `agents/<id>/02_prompt.md`
   * `agents/<id>/03_inputs/` (if applicable)
3. Place under `dag/cogniflow-ui/seed_pipelines/<name>/`.
4. Bump `app_version` in `dag/cogniflow-ui/config.json` (e.g. `1.0.0` → `1.1.0`).
5. Build and ship.

The next launch on every student machine will overlay the new and updated
pipelines. Same-named files in already-installed bundled pipelines will be
overwritten with the new content; user-created pipelines and runtime
artefacts are preserved.

### 3.3 Remove a bundled pipeline from future releases

Just delete its directory from `seed_pipelines/` and bump `app_version`.
Already-installed copies on student machines are not deleted — that would
destroy any student work in progress. They become "ordinary" pipelines from
the orchestrator's perspective.

### 3.4 Run against a different pipelines folder for development

Either edit `observer/config.json` and `configurator/config.json` to point
at a scratch folder, or set both env vars before launch:

```
ORCHESTRATOR_ROOT=/tmp/scratch-orch PIPELINES_ROOT=/tmp/scratch-orch/pipelines \
  venv/Scripts/python -m uvicorn app:app
```

The seeder will populate the scratch folder on first launch and write its
marker there.

### 3.5 Disable the Specialize (meta-prompt) feature

The recommended way is to set `"enable_specialize": false` in
`configurator/config.json`. This is a master switch: the "Generate" button is
never rendered (and the "claude CLI not found" warning is suppressed),
regardless of whether the CLI is installed or `claude_bin` is configured.

Alternatively (legacy path), leave `claude_bin` empty and ensure `claude` is
not on `PATH` (or set `CLAUDE_BIN=` empty). The UI detects this at startup,
sets the `claude_bin_found` template flag to `false`, and hides the
Specialize action — but in this case the "claude CLI not found" warning
banner is shown on the meta-prompts page. No code change required for either
approach.

## 4. Troubleshooting

| Symptom | Likely cause | Fix |
|---|---|---|
| Home page shows "No pipelines found" on first launch | `orchestrator_root` resolves to a directory without a `pipelines/` subfolder, so the seeder created `pipelines/` from scratch but the observer is reading a different path. | Verify `observer/config.json` and `configurator/config.json` resolve to the same on-disk folder. |
| Bundled pipeline edits don't propagate to a student | They are running an older `app_version` of the UI, or their marker was already at the new version (e.g. they tested a pre-release). | Bump `app_version` past their last-recorded value, or delete their marker. |
| Student created a pipeline named `01-writer-critic` and it keeps getting overwritten on UI upgrade | Name collision with a bundled pipeline. The seeder treats any directory named identically to a bundled one as a target for overlay. | Rename the student pipeline, or remove `01-writer-critic` from `seed_pipelines/` in the next release. |
| `[seed] skipped` printed every launch even after a version bump | `app_version` was bumped only in one place. The top-level `config.json` is the single source of truth. | Edit `dag/cogniflow-ui/config.json` (not the observer or configurator one). |
| Specialize button missing | Either `enable_specialize` is `false` in `configurator/config.json`, or `claude` CLI not found on PATH and `claude_bin` is empty. | Set `"enable_specialize": true` if disabled. Otherwise, install Claude CLI and either add it to PATH or set `claude_bin` explicitly. |
| 404 on `/pipelines/<name>/history` | `versioning: false` in `observer/config.json`. | Set it to `true` and restart. |

### 1. Fix: "Create Agent" returned `Agent 'new' not found`

**Symptom**
Clicking the **+ New Agent → Create Agent** button on the Agents tab of a blank
pipeline rendered an error page `Agent 'new' not found` and showed the agent
metadata form populated with display name "new".

**Root cause**
In `configurator/app.py`, FastAPI matches routes in registration order. The
generic `POST /pipeline/{name}/agent/{agent_id}` (metadata save) was registered
**before** `POST /pipeline/{name}/agent/new` (create). Submitting the create
form therefore matched the metadata-save route with `agent_id="new"`, which
called `fs.update_agent("new", …)` and returned `Agent 'new' not found`.

**Fix**
Reordered the agent routes in `configurator/app.py` so the literal-path route
is registered first:

```
POST /pipeline/{name}/agent/new            ← create (registered FIRST now)
GET  /pipeline/{name}/agent/{agent_id}     ← detail
POST /pipeline/{name}/agent/{agent_id}     ← save metadata
```

**Files changed**
- `configurator/app.py` — moved `agent_create` above `agent_detail` /
  `agent_save_meta`.

**Forward-port note**
Any future agent-scoped sub-route whose path segment is a literal (not a
captured `{agent_id}`) must be declared **before** the generic
`/agent/{agent_id}` handlers, or use a Path/regex constraint on `agent_id`
that excludes the literal.

### 2. UX: Sort agents by ID (NNN_ prefix) in the Topology view

**Request**
Agents in the Topology view should always be listed by their `NNN_` prefix,
i.e. `002_…` always precedes `003_…`.

**Implementation**
Sort happens in the template via Jinja2's `sort` filter. Because IDs use a
zero-padded 3-digit prefix (`NNN_name`), alphabetical sort yields the correct
numeric ordering. `selectattr('id')` defensively filters out malformed
entries.

**Files changed**
- `configurator/templates/partials/topology.html` — at the top of the file:

  ```jinja
  {% set raw_agents = pipeline.get('agents', []) if pipeline else [] %}
  {% set agents = raw_agents | selectattr('id') | sort(attribute='id') | list %}
  ```

**Forward-port note**
The SVG diagram in `dag_svg.build_svg` is generated server-side and may have
its own ordering — the change above only affects the agents table in the
Topology panel.

### 6. Audit: agent-type support across orchestrator v1 and observer v1

The user asked which of `orchestrator | worker | reviewer | synthesizer |
router | classifier | validator | summarizer` are actually executable by
the orchestrator and renderable by the observer, which require type-specific
implementations, and which fail gracefully.

#### TL;DR

| Type | Executable by orchestrator | Rendered by observer | Type-specific runtime code? |
| --- | :---: | :---: | --- |
| orchestrator | ✓ | ✓ | No — runs as a generic Claude call |
| worker       | ✓ | ✓ | No — runs as a generic Claude call |
| reviewer     | ✓ | ✓ | No — runs as a generic Claude call |
| synthesizer  | ✓ | ✓ | No — runs as a generic Claude call |
| classifier   | ✓ | ✓ | No — runs as a generic Claude call |
| validator    | ✓ | ✓ | No — runs as a generic Claude call |
| summarizer   | ✓ | ✓ | No — runs as a generic Claude call |
| **router**   | ✓ | ✓ | **Yes** — only if a `router.routes` block is set on the agent (see below) |

#### Why all eight types execute identically

Inspection of `dag/cogniflow-orchestrator/orchestrator/`:

- `core.py` walks DAG layers and dispatches each agent to `exec_agent`.
- `agent.py::exec_agent` reads `01_system.md` + `02_prompt.md`, calls
  the Claude CLI with `--system-prompt` + stdin, captures the JSON
  envelope, writes `05_output.md` / `05_usage.json`, then calls
  `_evaluate_router`.
- `validate.py::validate_pipeline` does **not** read `agent.type` — only
  `id`, `depends_on`, `router`, and per-agent `00_config.json`.

So `agent.type` is **not** consulted at runtime. The string is purely a
label used by the configurator UI to:

1. Pick which seed templates to copy in
   `fs.add_agent` →
   `prompt_templates/<type>/{01_system.md,02_prompt.md}` are read by
   `_read_prompt_template(atype, …)` (see `configurator/filesystem.py`).
2. Drive the type-badge colour on agent cards
   (`type-{{ agent.type or 'worker' }}`).
3. Feed the meta-prompt specialization (`⚙ Generate Prompts`) — the
   per-type `description.md` is sent to the meta-prompt pipeline.

#### Router is the one runtime-special type — but only conditionally

`agent.py::_evaluate_router` activates only when the agent's
`00_config.json` contains a `router` block, like:

```json
{ "router": { "routes": { "yes": ["003_branch_a"], "no": ["004_branch_b"] } } }
```

Behaviour:

1. After the agent runs, the orchestrator looks for `routing.json` in
   the agent's directory (the agent must produce this — typically by
   prompting Claude to emit a structured decision).
2. The decision is matched against `router.routes`. Targets under the
   chosen key stay live; targets under all other keys are marked
   bypassed (status = `bypassed`, event `agent_bypassed`).
3. If the decision is missing from `routes`, the orchestrator raises
   `RouterError` (a hard failure, not a graceful one).
4. If `routing.json` is missing, the orchestrator silently skips
   routing — the routes are not enforced.

> ⚠️ **The current configurator UI does not yet expose a `router.routes`
> editor.** A "router"-typed agent created through the UI runs identically
> to a "worker" — it calls Claude and exits, with no bypass effect on
> downstream agents. To get real routing semantics today you must hand-
> edit `agents/<router-id>/00_config.json` with a `router.routes` block,
> and the agent's prompt must produce a `routing.json`.

#### What happens for "missing-template" types

Each type listed in `agent_types` should have a folder under
`configurator/prompt_templates/<type>/`. The eight defaults all do.
If an operator adds a new value to `agent_types` without creating the
folder, the agent is created with **empty** `01_system.md` and
`02_prompt.md` (silent).

The orchestrator's `validate_pipeline` then catches the empty files
**only** if they are missing on disk; it does **not** flag empty
content. So:

- Type folder exists but files are empty → orchestrator runs the agent
  with an empty system prompt; Claude will be called with no
  guardrails, behaviour undefined. **Not** a graceful failure — runs
  but produces meaningless output.
- Type folder is missing entirely and `_read_prompt_template` returns
  `""` → same as above.
- The configurator's `⚙ Generate Prompts` button **does** fail
  gracefully if `description.md` or `01_system.md`/`02_prompt.md`
  template content is missing for the type — it returns a structured
  error in the agent detail panel without overwriting prompts.

#### Observer rendering

The observer (`dag/cogniflow-ui/observer/`) does **not** dispatch on
`agent.type` anywhere. The agent card (`partials/single_card.html`)
shows: id, description, status colour, duration, output bytes, cost,
optional context-token bar, and an Approve/Reject affordance when
`requires_approval` is true. All eight types render identically.

Router-related runtime events (`router_decision`, `agent_bypassed`)
**are** surfaced in the events feed (see
`templates/partials/events.html` — the colour code keys off
`'bypass'` substring), so a router with real routes will visibly
dim the bypassed agents.

#### Bottom line for the v1 stack

- **All eight types are executable.** The orchestrator runs them
  identically through the same Claude subprocess pipeline.
- **All eight types render in the observer.** No type-specific UI
  branches exist.
- **Only `router` has type-specific runtime semantics**, and only when
  the agent has a `router.routes` block in its `00_config.json` plus a
  prompt that produces `routing.json`. Without those, a "router"
  behaves like a worker.
- **Graceful failure** is limited: the orchestrator hard-raises
  `RouterError` on an unrecognised routing decision. Empty/missing
  prompt templates are *not* caught — the agent will run with
  effectively no system prompt. The configurator's prompt-specialization
  step is the only place that fails gracefully on missing template
  content.

### 7. Open enhancement: surface `static_inputs` in the configurator UI

**Context**
The orchestrator's only mechanism for feeding input to the first (root)
agent of a pipeline is the **`static_inputs`** array in that agent's
`agents/<id>/00_config.json`. Files listed there are resolved relative
to the pipeline directory, copied into `agents/<id>/03_inputs/static/`
at run time, and merged into `04_context.md` under an `# Input files`
section before Claude is invoked. See
`dag/cogniflow-orchestrator/orchestrator/context.py::collect_inputs` /
`assemble_context`.

Validation (`orchestrator/validate.py`) hard-fails the run if a path in
`static_inputs` does not exist on disk, so a missing or typo'd file is
caught up-front rather than silently producing an empty input section.

**Gap in v1 UI**
The configurator agent detail panel (`partials/agent_detail.html`)
exposes Display name, Type, Category, Timeout, Description, Labels,
Require approval, and Budget strategy — but **not** `static_inputs`.
Today the only ways to wire input for the root agent are:

1. **Files tab** — upload/create the input file under the pipeline
   directory, then create or hand-edit
   `agents/<root-id>/00_config.json` and add the `static_inputs` array.
2. Inline the input directly in `02_prompt.md` (only practical for
   short fixed inputs).

Both paths require operators to edit JSON by hand, which is error-prone
and inconsistent with how `budget_strategy` is already surfaced as a
first-class form field even though it also lives in `00_config.json`.

**Proposed enhancement (v2)**
Add a **Static inputs** field to the agent detail form — a textarea
or repeating row list of pipeline-relative paths — that is persisted
into `00_config.json["static_inputs"]` by the existing
`agent_save_meta` handler. Suggested behaviours:

- Accept one path per line; trim and drop blanks on save.
- Show, next to each row, a green/red dot indicating whether the file
  currently resolves under the pipeline directory (mirrors the
  validator's check).
- For convenience, allow a **Browse pipeline files…** button that opens
  the existing file browser scoped to the pipeline root, so operators
  can pick files instead of typing paths.
- Optionally: when an agent has no upstream `depends_on`, render the
  field with a slightly stronger affordance ("Inputs for this entry-
  point agent"), since this is the only knob on a root agent that
  actually reaches Claude beyond `02_prompt.md`.

**Backend touchpoints**
- `configurator/app.py::agent_save_meta` — read a new `static_inputs`
  Form field, normalise to a list of strings, and pass it through to
  `fs.update_agent` so it lands in the agent's `00_config.json` (today
  `update_agent` only writes to `pipeline.json`'s agent dict, so a small
  helper in `filesystem.py` to read/merge/write `00_config.json`
  alongside the pipeline-level update is needed).
- `configurator/filesystem.py` — add `update_agent_runtime_config(name,
  agent_id, updates)` writing to `agents/<id>/00_config.json` atomically,
  preserving unknown keys (so future fields like `cli_flags`,
  `max_retries`, `retry_delays_s` remain editable).

**Forward-port note**
The same control would also benefit non-root agents that need extra
shared fixtures alongside upstream outputs. The orchestrator already
supports `static_inputs` on every agent, not just roots — the UI
should not gate the field by depth.

### 8. Reference: the "shared fixtures" pattern with `static_inputs`

This section explains a usage pattern of the `static_inputs` mechanism
described in section 7. It is reference material for operators and for
future UI work — it does not describe a code change in this session.

#### Problem this pattern solves

Some inputs aren't agent-specific — they are reference material that
several agents in the pipeline must anchor on. Typical examples:

- A project brief that the researcher, analyst, and reviewer all need.
- A schema or API spec that several worker agents must conform to.
- A glossary, style guide, or set of constraints applied across the
  pipeline.

Without `static_inputs`, the operator has only two workable options, and
both have downsides:

1. **Paste the content into each agent's `02_prompt.md`.** The same
   paragraph then lives in N files. When the brief changes the operator
   must update N files and keep them in sync — easy to forget, easy to
   drift.
2. **Daisy-chain the fixture through `depends_on`.** A "publisher" agent
   emits the brief as `05_output.md`, and every consumer depends on it.
   This works mechanically but lies about the dependency graph (a static
   file is not a real upstream computation), and it forces an extra
   agent run just to forward bytes.

#### How `static_inputs` solves it

The fixture lives once on disk under the pipeline directory. Each agent
that needs it lists the same pipeline-relative path in its
`00_config.json` `static_inputs`:

```
pipelines/my_pipeline/
  pipeline.json
  fixtures/
    brief.md              ← lives here ONCE
    api_schema.json
  agents/
    001_researcher/00_config.json   →  {"static_inputs": ["fixtures/brief.md"]}
    002_analyst/00_config.json      →  {"static_inputs": ["fixtures/brief.md", "fixtures/api_schema.json"]}
    003_reviewer/00_config.json     →  {"static_inputs": ["fixtures/brief.md"]}
```

#### What "copied per-agent at run time" means

The file is *stored* once on disk. Each time the orchestrator launches
an agent, `collect_inputs` reads
`pipelines/my_pipeline/fixtures/brief.md` and writes a fresh copy into
that agent's local working area:

```
agents/001_researcher/03_inputs/static/brief.md
agents/002_analyst/03_inputs/static/brief.md
agents/003_reviewer/03_inputs/static/brief.md
```

`assemble_context` then fences each file into the agent's
`04_context.md`. Claude sees the brief in the agent's prompt as if it
had been pasted inline — but the source of truth is the single file
under `fixtures/`.

#### Why the per-agent copy exists (rather than reading in-place)

1. **Run-time isolation.** Each agent's `03_inputs/` is a self-contained
   snapshot of what was true when *that* agent started. If the operator
   edits `fixtures/brief.md` mid-run, downstream agents do not
   accidentally see a different version than upstream agents — each
   carries its own snapshot.
2. **Reproducibility / audit.** Inspecting
   `agents/002_analyst/03_inputs/static/brief.md` after a failed run
   shows the exact bytes Claude was given for that agent, even if the
   source file later changed.
3. **Symmetry with upstream outputs.** Static inputs and upstream
   `from_<dep>.md` files end up side-by-side in the same `03_inputs/`
   folder, so the context-assembly step treats them uniformly.

#### Net effect

One source of truth on disk, with the convenience of inline content in
every agent's context, plus a per-run audit trail. Edit the fixture
once; every agent that lists it picks up the new version on the next
run — no per-agent file edits required.

#### Implication for the v2 UI work in section 7

The "Static inputs" form field proposed in section 7 should make this
pattern easy to express:

- Allow the same path to appear in multiple agents' lists without
  warning (it is intentional, not a duplicate).
- The proposed **Browse pipeline files…** picker should default to a
  shared `fixtures/` (or `inputs/`) directory if one exists, since
  shared fixtures are the dominant use case.
- Documentation surfaced near the field should explicitly call out
  this pattern so operators do not reach for the prompt-paste or
  dependency-injection workarounds.

rather than metadata-only — so downstream agents have something real to

#### Layout


#### Intentional content for downstream stages

The fixture is engineered to exercise specific downstream agents, not
just to fill bytes:

- **PII scrubber (`002_pii_scrubber`)** — records contain emails
  (`sarah.morgan@example.com`, `maria.lopez+billing@example.es`,
  `devops-lead@example.com`, `jordan.kim@example.org`), phone numbers
  (`+1-415-555-0149`, `0049-30-555-0188`), and account IDs
  (`778-A4-991132`, `442-EU-77310`).
- **Quality scanner (`003_quality_scanner`)** — `train_03.jsonl` has
  one empty-text record, one with `created_at: "not-a-real-timestamp"`,
  one with mismatched language tag (Turkish text labelled `en`), one
  with a duplicated `id` (collision with `train_00` row 1), and one
  with another wrong-language tag (English text labelled `de`).
- **Schema validator (`004_schema_validator`)** — `schema.json` defines
  enums for `language`, `source_channel`, `label`. The Turkish-text
  record's language tag (`en`) lies outside the observed-language note
  in the README; the duplicated id violates uniqueness expectations.
- **Train/eval split (`005_train_split`)** — class distribution across
  `train_00`..`train_03` is intentionally uneven (positive ≈ 42%,
  neutral ≈ 31%, negative ≈ 21%, mixed ≈ 6%) so the splitter has
  something to balance.
- **Eval suite (`008`–`012`)** — the holdout shard is unlabelled, so
  evaluators must declare label assumptions explicitly rather than
  reading them from the records.

#### How the orchestrator surfaces the fixture

`fs.collect_inputs` copies every file listed in `static_inputs` into
`agents/001_dataset_pull/03_inputs/static/`, **flattened by basename**
(see `orchestrator/context.py::collect_inputs`, `dst = static_dir /
src.name`). The fixture's basenames are all unique so no collisions
occur.

`fs.assemble_context` then writes them into `04_context.md` under an
`# Input files` section, each fenced by extension-derived language hint
(`.md` → `markdown`, `.json` → `json`, `.jsonl` → `""`). The agent
sees them inline in its prompt and emits the manifest from the
authoritative listing in `dataset_pull_input.json`.

#### Wiring downstream agents

Downstream agents that need to read the *raw data* (not just the
manifest) follow the **shared-fixtures pattern** documented in
section 8: each agent's `00_config.json` declares the same
`fixtures/raw/...` paths in its own `static_inputs`. The fixture lives
once on disk, every agent that lists it gets a per-run snapshot in
its `03_inputs/static/`. Example for `002_pii_scrubber`:

```json
{
  "static_inputs": [
    "fixtures/raw/shards/train_00.jsonl",
    "fixtures/raw/shards/train_01.jsonl",
    "fixtures/raw/shards/train_02.jsonl",
    "fixtures/raw/shards/train_03.jsonl",
    "fixtures/raw/holdout/holdout_blind.jsonl"
  ]
}
```

The dependency graph still flows through `depends_on` — `002` depends
on `001` so it receives `001`'s manifest as `from_001_dataset_pull.md`
in its context — but the *bytes* are duplicated in via `static_inputs`.
This is the correct pattern under the v1 architecture (Claude cannot
read disk; everything must arrive via context).

#### Expected manifest output from `001_dataset_pull`

A single JSON object (no fences) with:

- `source.bucket = "acme-training-data-raw"`,
  `source.prefix = "customer-feedback-classifier/v3/2026-04/"`
- `staging_location =
  "/var/cogniflow/staging/20-model-release-dag/run-2026-04-29/raw/"`
- `pulled_at = "2026-04-29T10:42:00Z"`
- `object_count = 9`
- `total_bytes = 19150`
- `objects[]` with 9 entries, each carrying `key`, `staged_path`,
  `size_bytes`, and `etag` matching the table above. `staged_path` is
  derived by stripping the `source.prefix` from `key` and prepending
  `staging_location`.
- `status = "complete"`

#### Caveats

- **Staging is symbolic.** `staging_location` is `/var/cogniflow/...` —
  that path is not actually created on disk. It exists only as a
  string in the manifest for downstream agents to reference. Real
  bytes for downstream agents arrive via shared `static_inputs` (see
  above), not by reading `staging_location`.
- **Total payload is ~19 KB**, well under
  `model_context_limit = 180000` tokens (~720 KB), so the entire
  dataset fits in a single agent's context. For larger fixtures the
  shared-fixtures pattern still applies but per-agent budget strategy
  (`hard_fail` / `auto_summarise` / `select_top_n`) becomes relevant.
- **Schema and labels are bundled in fixtures/raw/**. They are
  intentionally part of the *raw* export, not synthesized by the
  pipeline, so `004_schema_validator` is validating the data against
  the schema *that shipped with it* — same as a real S3 export would.

#### Forward-port note

Future revisions should consider:

- Adding a small Python helper alongside the fixture
  (`fixtures/recompute_digests.py`) so editors can regenerate
  `manifest_source.json` and the `dataset_pull_input.json` etag/size
  fields after editing any raw file. Right now the digests are
  hand-maintained and will drift if anyone edits a `.jsonl`.
- Wrapping the shared-`static_inputs` boilerplate in a v2 UI affordance
  ("Use shared fixtures from this directory" — pulls all matching
  paths into the selected agent's `static_inputs`).
- Optionally splitting the fixture into a `fixtures/clean/` and
  `fixtures/raw/` so agents that should consume already-scrubbed data
  (eval / trainer) can switch sources without rewriting the pipeline.

### 10. Feature: `enable_specialize` master switch in `config.json`

**Request**
The configurator's "⚙ Generate" (a.k.a. Specialize) button was previously
gated only on whether the `claude` CLI could be located on the host. The
existing logic (`config.py::Settings.claude_bin`) returns the override
`config["claude_bin"]` *or* falls back to `_find_claude()`, which searches
`CLAUDE_BIN`, the PATH (`shutil.which("claude.cmd"|"claude.exe"|"claude.bat"|"claude")`)
and platform-specific candidate paths (`%LOCALAPPDATA%\Programs\claude\…`,
`/opt/homebrew/bin/claude`, etc.). On any developer machine that has the
CLI installed for unrelated reasons the button therefore always appears,
even when `claude_bin` is left empty in `configurator/config.json`. There
was no clean "feature off, regardless of CLI presence" knob.

**Implementation**
Added a new boolean setting `enable_specialize` (default `true`) parsed
exactly like `cfg_readonly` (accepts JSON bool, JSON string `"true"/"false"`,
or env-var override `ENABLE_SPECIALIZE=…`). Gating decisions are now:

| `enable_specialize` | `claude_bin` resolved | Generate button | "CLI not found" warning | Meta Prompts entry |
| :---: | :---: | :---: | :---: | :---: |
| `true`  | found      | shown    | hidden  | shown |
| `true`  | not found  | hidden   | shown   | shown |
| `false` | (any)      | hidden   | hidden  | hidden |

Specifically:

1. **Templates** — the agent-card and topology-row Generate buttons now
   gate on `claude_bin_found and enable_specialize` instead of just
   `claude_bin_found`. The "claude CLI not found" warning banner on the
   meta-prompts page gates on `enable_specialize and not claude_bin_found`,
   so it no longer shouts about a missing CLI when the feature is
   intentionally disabled.
2. **Sidebar** — `prompt_templates.html` wraps the entire **Meta Prompts**
   sidebar entry (label + link) in `{% if enable_specialize %}`. When the
   switch is off, only the "Agent Types" section is visible.
3. **Route hardening** — `prompt_templates_list` 303-redirects
   `?selected=_meta` to `/prompt-templates` when `enable_specialize` is
   false, so the meta editor cannot be reached even by direct URL.
4. **Template context** — every render path that already passed
   `claude_bin_found` now also passes `enable_specialize`. Six sites in
   `app.py` were updated (the `pipeline_view` render, `_topology_partial`,
   the `prompt_templates_list` render, and the three error-branch renders
   in `prompt_template_new` / `prompt_template_meta_save` /
   `prompt_template_save`).

**Files changed**
- `configurator/config.py` — new `enable_specialize: bool` field on
  `Settings` (parsed identically to `cfg_readonly`).
- `configurator/config.json` — new `"enable_specialize": true` key.
- `configurator/app.py` — added `"enable_specialize": settings.enable_specialize`
  to all six template contexts that already passed `claude_bin_found`;
  added the `_meta` redirect guard at the top of `prompt_templates_list`.
- `configurator/templates/partials/agents_tab.html` — Generate button
  gated on `claude_bin_found and enable_specialize`.
- `configurator/templates/partials/topology.html` — same.
- `configurator/templates/prompt_templates.html` — Meta sidebar entry
  wrapped in `{% if enable_specialize %}`; "CLI not found" warning
  banner gated on `enable_specialize and not claude_bin_found`.
- `docs/instruction.md` — new row in the config table at §2.3, rewritten
  §3.5 recipe ("Disable the Specialize feature"), updated troubleshooting
  row at §4.

**Smoke checks performed**
- `python -c "from configurator.config import settings; …"` →
  `enable_specialize=True` (config default) and
  `enable_specialize=False` under `ENABLE_SPECIALIZE=false`.
- With `"enable_specialize": false` in `config.json`,
  `list_prompt_templates()` returns the Agent-Types only, and the
  resolved `claude_bin` is irrelevant to the UI gating.

**Forward-port note**
The underlying `POST /pipeline/<x>/agent/<y>/specialize` and
`POST /prompt-templates/_meta/<filename>` endpoints are **not** gated
server-side — only the UI surface is. A direct curl/HTTP call could
still trigger specialization or save meta prompts when the feature is
"disabled". For defense-in-depth (e.g. shipping to operators where
`enable_specialize=false` must be a hard kill rather than a UI hint),
add a `if not settings.enable_specialize: return HTMLResponse(…, status_code=403)`
guard at the top of those route handlers. Left as a follow-up because
the immediate requirement was UI-only.

### 11. Fix: prompt-templates page listed agent-type folders not in `agent_types`

**Symptom**
On the **/prompt-templates** page, the left sidebar showed types like
`orchestrator` and `router` even though `configurator/config.json`'s
`agent_types` list explicitly excluded them
(`["worker", "reviewer", "synthesizer", "classifier", "validator", "summarizer"]`).
Stale folders left over on disk leaked into the UI as if they were
configured types.

**Root cause**
`filesystem.py::list_prompt_templates()` enumerated every directory under
`settings.prompt_templates_dir`, excluding only the reserved `_meta`
folder and dotfiles. It never consulted `settings.agent_types`, so any
directory present on disk became a sidebar entry — including legacy
folders for types since removed from the config.

**Fix**
Filter the directory walk by `settings.agent_types`:

```python
allowed = set(settings.agent_types)
…
if d.name not in allowed:
    continue
```

Now the listing reflects only types that are *both* configured and have
a folder on disk. Folders for types removed from `agent_types` remain on
disk untouched (no destructive cleanup), but they no longer appear in
the UI.

**Files changed**
- `configurator/filesystem.py` — `list_prompt_templates()` now filters
  by `settings.agent_types`.

**Smoke check performed**
- Before: on-disk dirs `[_meta, classifier, orchestrator, reviewer, router,
  summarizer, synthesizer, validator, worker]` →
  `list_prompt_templates()` returned all 8 non-meta entries (including
  `orchestrator`, `router`).
- After: same on-disk state →
  `list_prompt_templates()` returns the 6 in `agent_types`
  (`classifier, reviewer, summarizer, synthesizer, validator, worker`).
  `orchestrator` and `router` folders are still on disk but no longer
  listed.

**Forward-port note**
This change makes `agent_types` the authoritative list everywhere in
the UI: both the agent-create dropdowns (since section 5) and the
prompt-templates sidebar. Adding a new value to `agent_types` still
needs the matching `prompt_templates/<type>/` folder on disk, otherwise
the type appears in the dropdowns but has no entry in the prompt-
templates sidebar (so its starter prompts can't be edited via the UI,
only via filesystem).

### 12. Reference: Pipeline templates vs. prompt templates

Both surfaces are reachable from the same bottom-of-sidebar block in
`base.html:62-63`. They share the word "templates" but are unrelated
mechanisms with different scopes, storage locations, and lifecycles.
This section documents the contract for forward-porting.

#### Side-by-side

| Aspect | 📋 Pipeline Templates (`/templates`) | 🧩 Prompt Templates (`/prompt-templates`) |
| --- | --- | --- |
| Scope | Whole-pipeline starter bundle | Per-agent-type starter prompt pair (+ `description.md`) |
| Storage | `pipelines_root/templates/<tname>/` (sibling to user pipelines) | `configurator/prompt_templates/<type>/` (inside the configurator package) |
| Created by | `filesystem.py::save_as_template()` (`Save as template…` action on a pipeline) | Hand-edited or via `prompt_template_new` on the prompt-templates page |
| Consumed when | A **new pipeline** is created from a template (`filesystem.py::_apply_template`) | An **agent** of type `X` is added to any pipeline (`filesystem.py::_read_prompt_template` reads `prompt_templates/X/{01_system.md,02_prompt.md}`) |
| Per-pipeline or shared? | Shared library, one entry per template name | Globally available across **all** pipelines — same `prompt_templates/` directory feeds every pipeline's agent-create flow |
| Live or snapshot? | **Snapshot.** Files are deep-copied at create-time. Editing the template later does **not** affect pipelines already created from it. | **Snapshot.** Files are copied into the agent's folder at create-time. Editing the template later does **not** affect existing agents — only new agents (or agents re-tailored via ⚙ Generate) pick up the new content. |
| Includes the meta-prompts? | No | Yes — the reserved `_meta/` directory holds the system + task prompts that drive the Specialize ("⚙ Generate") feature. Listed separately from `agent_types` in the sidebar; hidden entirely when `enable_specialize=false` (see section 10). |

#### What "do they apply to all pipelines?" means in practice

- **Pipeline Templates — no, not in any live sense.** They are clone-
  on-create blueprints. After a new pipeline is created from a template,
  the new pipeline is an independent copy with no back-reference to the
  template; future template edits will not flow into it.
- **Prompt Templates — globally available, but still clone-on-create
  per agent.** The same library is offered to every pipeline, so the
  starter prompts you edit at `/prompt-templates/worker` are what *every*
  newly-added `worker` agent in *every* pipeline starts from. But once
  an agent has been created, its `01_system.md` and `02_prompt.md` are
  its own copies in `agents/<id>/`, and editing the type's template will
  not change the existing agent's prompts. To re-pull from the template
  you'd either delete-and-recreate the agent or run the ⚙ Generate
  meta-prompt pipeline on it.

#### Why the distinction matters

Two practical consequences for operators:

1. **Editing a prompt template is "free" mid-project** — it changes
   nothing already in flight, so you can iterate on type defaults
   without disturbing live pipelines. The downside is the inverse: a
   bug fix in `prompt_templates/worker/01_system.md` does not propagate
   to existing workers; each must be re-edited or re-specialized.
2. **Saving a pipeline as a template is a *fork point*, not a *binding*** —
   the act of "Save as template" snapshots the current pipeline; subsequent
   edits to the source pipeline do not flow into the template, and edits
   to the template do not flow into pipelines previously cloned from it.
   Treat templates as labelled fork points, not as living parents.

#### Files of interest

- Pipeline templates code path:
  `filesystem.py::list_templates`, `save_as_template`, `_apply_template`;
  template UI: `templates/templates.html`.
- Prompt templates code path:
  `filesystem.py::list_prompt_templates`, `_read_prompt_template`,
  `read_meta_prompt`; UI: `templates/prompt_templates.html` and the
  per-agent copy logic in `add_agent` (which calls `_read_prompt_template`).

### 14. Fix: `_dir` field in `pipeline.json` was an absolute path

**Symptom**
Inspecting `pipeline.json` on disk (or in the JSON editor) showed an
absolute path under the `_dir` key, e.g.
`"_dir": "C:\\Users\\edatlab\\…\\pipelines\\01-writer-critic"`. This
pollutes pipeline files with a non-portable, host-specific value that
breaks copying the pipeline to another machine and bloats the JSON
editor with information the operator did not write.

**Root cause**
`filesystem.py::get_pipeline()` decorates the loaded dict with
`data["_dir"] = str(d)` where `d = settings.pipelines_root / name` —
so reading a pipeline injects the absolute path. Most write paths
read from disk fresh and never serialise `_dir`, but
`graph_orientation_toggle` (and any code path that calls
`fs.get_pipeline()` and feeds the result through
`fs.write_pipeline_json(...)`) does serialise it back. Every time the
operator toggles orientation, `_dir` is persisted with whatever
absolute value the host happens to have.

A grep for the literal `"_dir"` key across the repo confirms **no
code reads it** — it is purely informational metadata. So the fix is
free: change the value, no consumer needs to be updated.

**Fix**
`filesystem.py:107` now writes `data["_dir"] = name` — i.e. just the
pipeline directory name, which is the directory's path relative to
`settings.pipelines_root` (and equivalently relative to the
orchestrator install once `orchestrator_root + "/pipelines"` resolves
to `pipelines_root`). Pipelines that already have an absolute `_dir`
on disk will be normalised to the relative form on the next save (the
in-memory dict from `get_pipeline` now carries the relative form, so
any subsequent persist writes the relative value).

**Files changed**
- `configurator/filesystem.py` — `get_pipeline()` writes `_dir = name`
  instead of `_dir = str(d)`.

**Smoke check performed**
```
python -c "from configurator import filesystem as fs; \
  print(fs.get_pipeline(fs.list_pipelines()[0]['name'])['_dir'])"
## → '01-writer-critic'   (was 'C:\\…\\pipelines\\01-writer-critic')
```

**Forward-port note**
- `_dir` is unread; consider deleting it entirely in a future cleanup
  pass. Kept for now because the v1 JSON editor surfaces the field
  and operators may have come to expect it.
- If `_dir` ever becomes load-bearing, do **not** trust the persisted
  value: always recompute from `pipelines_root / name`. The persisted
  copy can drift if a pipeline directory is renamed on disk (no UI
  rename today; section 15 only covers agent rename, not pipeline
  rename).

### 15. Feature: rename agent (id) from the Edit Agent modal

**Request**
When you create a pipeline from a Pipeline Template (section 12), the
agents inherit the template's IDs (e.g. `001_researcher`,
`002_analyst`) and display names. The v1 Edit Agent modal exposed
display name, type, category, description, depends-on, timeout, and
require-approval — but the **Agent ID** field was rendered with
`disabled` and a hidden mirror field. So the only way to change a
template-supplied id was to edit `pipeline.json` by hand and rename
`agents/<old>/` on disk; the UI offered no rename affordance.

**Implementation**
Three coordinated changes:

1. **`filesystem.py::rename_agent(pipeline_name, old_id, new_id)`** —
   new function next to `update_agent`. Validates the new id against
   the same `^\d{3}_[a-z0-9_]+$` pattern the Add Agent form uses;
   no-ops when `old_id == new_id`; refuses if the new id collides
   with an existing agent or if `agents/<new_id>/` already exists on
   disk. On success it:
   - `Path.rename(old_dir, new_dir)` — atomic same-filesystem move
     of the agent's workspace directory.
   - Rewrites the `id` field of the matching agent in
     `pipeline.json["agents"]`.
   - Rewrites every `depends_on` list across all other agents,
     replacing `old_id` with `new_id` element-wise.
   - Rewrites every `edges[i].from` / `edges[i].to` that referenced
     `old_id`.
   - Saves a snapshot to the version history with message
     `Rename agent <old> -> <new>`.
2. **`app.py::topology_update_agent`** — now accepts a `new_agent_id`
   form field. If non-empty and different from the URL-supplied
   `agent_id`, calls `fs.rename_agent` first and tracks the resulting
   `effective_id`. The subsequent `fs.update_agent` call uses
   `effective_id`, so metadata edits and id renames are applied in a
   single PUT round-trip. If the rename fails (validation, collision,
   filesystem error), the route returns the topology partial with
   the error banner and skips the metadata update.
3. **`templates/partials/topology.html`** — the Edit Agent modal's
   Agent ID input is no longer `disabled`. It now carries
   `name="new_agent_id"`, the same `pattern="\d{3}_[a-z0-9_]+"`
   constraint as the Add Agent modal, and a small helper line
   warning the operator that renaming moves `agents/<id>/` on disk
   and rewrites `depends_on` and edge references.

The original `_agent_id` hidden field still carries the **old** id
and is what the URL of `PUT /pipeline/{name}/graph/agent/{agent_id}`
keys on, so the route always knows which agent to look up before
applying the rename. The frontend `submitEditAgent` in
`static/app.js` did not need to change — `FormData(form)` picks up
the new field automatically.

**Files changed**
- `configurator/filesystem.py` — added `_AGENT_ID_RE` and the
  `rename_agent` function; added `import re`.
- `configurator/app.py` — `topology_update_agent` now accepts
  `new_agent_id` and routes through `rename_agent` when supplied.
- `configurator/templates/partials/topology.html` — Agent ID input
  in the Edit Agent modal is now editable with the format pattern.

**Smoke checks performed**
```
python -c "from configurator import filesystem as fs; \
  print(fs.rename_agent('does-not-exist', '001_a', '001_a')); \
  print(fs.rename_agent('does-not-exist', '001_a', 'BadID')); \
  print(fs.rename_agent('does-not-exist', '001_a', '001_renamed'))"
## → (True, 'ok')                          ← no-op same id
## → (False, "New agent id 'BadID' must …") ← format reject
## → (False, 'pipeline.json not found')    ← missing pipeline
```

**Forward-port note**
- The matching display-name field (`agent_name`) was already editable
  before this change; the rename feature only addresses the ID. Both
  surfaces are now decoupled: an operator can rename the id, the
  display name, or both independently.
- Renaming an agent does **not** rewrite the agent's own prompt
  contents, the stored versions in `.configurator/history/`, or any
  output files left from previous runs. If those reference the agent
  by id (in narrative text rather than structural fields), the
  references are not auto-updated. This is intentional —
  re-specialisation via ⚙ Generate is the right tool to refresh
  prompt text after a rename.
- The Add Agent modal already provides a Display name field; the
  paired pattern is "rename id + rename display name" rather than
  "edit display name only", since template-cloned agents typically
  need both adjusted together.

### 16. UX: New Pipeline modal — sanitise directory name and auto-populate Display Name

**Request**
The Directory name field in the **+ New Pipeline** modal already
carried `pattern="[a-z0-9_-]+"`, but the validation only fires on
form submit. Users could still type spaces and uppercase letters and
only learn the constraint when their submission was rejected. They
also had to manually retype the same characters into Display Name to
get a human-readable title, which is redundant work.

**Implementation**
Two coordinated handlers in `templates/index.html`:

1. **Directory name** input gains an `oninput` handler
   (`onNewPipelineDirInput(this)`) that, on every keystroke:
   - Lowercases the value.
   - Replaces runs of whitespace with a single hyphen.
   - Strips any character that isn't `[a-z0-9_-]`.
   - Writes the cleaned value back to the input and best-effort
     restores the caret position.
   The HTML `pattern` attribute is kept as a belt-and-braces server-
   side guarantee.
2. **Display Name** input gains
   `oninput="this.dataset.touched='1'"` — a one-time flag that
   marks the field as user-edited.
3. The Directory name handler also auto-populates Display Name **only
   when** the touched flag is absent: it derives a title-cased
   string by replacing `-`/`_` runs with single spaces and
   upper-casing each word's first letter. The moment the user types
   anything in Display Name, the touched flag latches and the auto-
   populate stops clobbering their edits.

So the typical flow is: type "AI Sentiment Pipeline" in Directory
name → input becomes `ai-sentiment-pipeline`, Display Name becomes
"Ai Sentiment Pipeline" automatically. The user can then edit
Display Name (e.g. capitalise "AI") and from then on Directory name
edits never overwrite their wording.

**Files changed**
- `configurator/templates/index.html` — added `id`, `oninput` on
  Directory name and Display Name inputs; added a small `<script>`
  block defining `onNewPipelineDirInput`.

**Forward-port note**
- The same UX would apply to the Add Agent modal (`topology.html`),
  whose Agent ID field has a stricter `\d{3}_[a-z0-9_]+` pattern.
  Auto-populating from the prefix is less natural there because the
  `NNN_` part is computed (next free triple), but a similar
  sanitiser on the suffix could be added if the request comes up.
- The script is intentionally inline in `index.html` rather than
  added to `static/app.js` because the handler is scoped to that
  modal only and is shorter than the import boilerplate would be.
  Re-evaluate if a second modal needs the same logic.

### 17. Fix: pipeline-delete left a half-deleted pipeline (agents gone, `pipeline.json` retained)

**Symptom**
Clicking the top-right **🗑 Delete** button on the pipeline view (or
the corner `✕` on a pipeline card on the home page) appeared to do
nothing visible — no error banner, no redirect to `/configurator` —
but inspecting the filesystem afterwards revealed the pipeline
directory still in place with `pipeline.json` (and the directory's
sidebar entry) preserved while the **`agents/` subtree had been
removed**. Re-clicking Delete repeated the same partial behaviour.

**Root cause**
`filesystem.py::delete_pipeline()` used `shutil.move(str(d), str(dest))`.
On Windows, when `os.rename(src, dst)` fails (typically because some
file inside `src` is held open by another process — orchestrator,
editor, file watcher), `shutil.move` falls back to `copytree(src, dst)`
followed by `shutil.rmtree(src)`. `rmtree` walks bottom-up, removing
files in the deepest directories first. So `agents/<id>/01_system.md`,
`02_prompt.md`, etc. — and ultimately `agents/` itself — are removed
*before* the walk reaches the locked file at the top level. When the
walk fails on the locked file (`pipeline.json`, `.state/...`, or
similar), `rmtree` raises mid-walk and the source ends up with
everything except the locked file deleted.

The exception then propagated out of the `delete_pipeline` route in
`app.py` without being caught, so the user did not see a styled error
banner — just whatever FastAPI's default exception handler returned
for the unhandled `OSError`. The browser displayed a 500 page that did
not look like a normal error banner, and the user perceived it as "no
visible response", consistent with their report.

**Fix**
Two coordinated changes that together guarantee atomic-or-nothing
delete semantics with clear error feedback:

1. **`filesystem.py::delete_pipeline()`** rewritten to:
   - Return `tuple[bool, str]` instead of `bool`, so the route can
     surface the error message.
   - Use `Path.rename(src, dst)` (a thin wrapper around the OS-level
     atomic rename — `MoveFileEx` on Windows, `rename(2)` on POSIX)
     instead of `shutil.move`. The atomic rename either succeeds for
     the entire directory or fails leaving the source untouched. There
     is no copy+delete fallback to half-finish.
   - Catch `OSError` at three points (trash-mkdir, stale-trash rmtree,
     final rename / non-trash rmtree) and return a descriptive
     message naming the path and recommending the operator close any
     process holding the directory open.
2. **`app.py::delete_pipeline()` route** now unpacks `(ok, msg)` and,
   on failure, returns an `HTMLResponse` with a styled error banner
   identifying the pipeline and the underlying reason. The redirect to
   `/configurator` only fires on success.

After the fix:
- A successful delete behaves exactly as before (atomic move into
  `<pipelines_root>/.trash/<name>/`, then 303 to home).
- A failed delete leaves the pipeline directory **completely
  untouched** and shows the operator a banner like
  `Failed to delete pipeline 'foo': Could not move pipeline to
  trash: [WinError 32] The process cannot access the file because
  it is being used by another process: '...'. Close any process
  (orchestrator, editor, file watcher) holding files in
  '...\\foo' open and try again.`
- No partial state.

**Files changed**
- `configurator/filesystem.py` — `delete_pipeline` now returns
  `tuple[bool, str]`, uses atomic `Path.rename`, wraps each
  filesystem operation in `try/except OSError`.
- `configurator/app.py` — `delete_pipeline` route unpacks the tuple
  and returns a 500 with a banner on failure, redirect on success.

**Smoke check performed**
```
python -c "from configurator import filesystem as fs; \
  print(fs.delete_pipeline('does-not-exist'))"
## → (False, "Pipeline 'does-not-exist' does not exist on disk.")
```

**Recovering the half-deleted pipeline from the original incident**
The pipeline that hit the original bug is in a corrupted state:
`pipeline.json` still references agent ids whose directories no
longer exist. To clean up, either:

1. **Delete the now-empty pipeline directory from disk** — the new
   Delete button will succeed cleanly (the source has no descendants
   to lock), and the trash entry will be removed. Or:
2. **Re-add the agent directories** — under `agents/<id>/` create
   empty `01_system.md` and `02_prompt.md`. The pipeline becomes
   editable again, but its prompts are blank. Use ⚙ Generate to
   re-specialise.

Option (1) is the right answer in almost every case unless the
operator has live work in `pipeline.json` they want to preserve.

**Forward-port note**
- The atomic-rename strategy means deletes across mount points
  (e.g. `pipelines_root` on `C:` and `cfg_trash_dir` on `D:`) will
  fail with `OSError`, since `rename` cannot cross filesystems. If
  this becomes a real configuration, layer in a one-shot
  `copytree → rmtree(ignore_errors=True)` fallback **only after**
  the atomic rename fails — but keep the "all-or-nothing" guarantee
  by writing to a temp directory first and renaming into place.
- The same Windows-locked-file class of bug can affect other write
  paths (`save_as_template`, `restore_version`). If a similar
  symptom appears there, prefer atomic rename over copy+delete and
  surface errors rather than swallow them.

### 18. Fix: text-cursor in the file/prompt editors was offset half a character to the left

**Symptom**
In the configurator's file editor and prompt editor (and the observer's
history file editor), the visible text caret did not sit between the two
characters where the next typed character would be inserted. Instead it
sat **inside the body of the previous letter** — about half a character
to the left of the correct position. Typing made characters appear ~5 px
to the right of the caret.

**Root cause**
The editor uses a "mirror-overlay" layout: a transparent
`<textarea class="hv-editor">` sits on top of a syntax-highlighted
`<pre class="hv-editor-mirror"><code></code></pre>`. The textarea draws
the caret; the mirror draws the colored glyphs. For the caret to align
with the rendered glyphs, the two layers must have **identical** text
origins.

The shared CSS rule (configurator `style.css:583-599`, observer
`app.css:959-974`) gives `.hv-editor`, `.hv-editor-mirror`, and
`.hv-viewer-pre` the same `padding: 10px 12px` (configurator) /
`padding: 14px 16px` (observer). What it did *not* normalise is the
**inner `<code>` element inside the mirror's `<pre>`**, which inherits
the user-agent default `padding: 1px 5px` on `<code>`.

Net effect: the textarea text origin = `12 px` (padding-left). The
mirror text origin = `12 + 5 = 17 px`. The mirror's characters were
drawn 5 px to the right of where the textarea positioned its caret, so
the caret visually fell inside the previous glyph.

A Playwright check confirmed the offset numerically:

| measurement | x (px) |
| --- | ---: |
| textarea caret X (computed via canvas `measureText`) | 50.62 |
| mirror's same-character X (Range over the mirror's text) | 55.61 |
| **caret − mirror** | **−4.99** |

After the fix, the same measurement reads `+0.01 px`.

**Fix**
Add `padding: 0; margin: 0;` to the `.hv-editor-mirror code` rule in
both stylesheets so the inner `<code>` no longer contributes its UA
padding to the mirror's text origin.

```css
.hv-editor-mirror code {
  font: inherit; color: inherit; background: transparent;
  white-space: inherit; word-break: inherit; display: block;
  padding: 0;
  margin: 0;
}
```

**Files changed**
- `configurator/static/style.css` — rule at `.hv-editor-mirror code`
  (lines 611-620): added `padding: 0; margin: 0;`.
- `observer/static/app.css` — rule at `.hv-editor-mirror code`
  (lines 1000-1009): added `padding: 0; margin: 0;`.

**Smoke check performed**
Started the dev server and ran a Playwright probe that placed the caret
at index 5 of the textarea content, then measured the textarea caret X
(via `getBoundingClientRect` + canvas `measureText`) against the same
character's X in the mirror (via a `Range` over the highlighted text
nodes). Offset before the fix: −4.99 px. Offset after: +0.01 px.

**Forward-port note**
- The same trap will reappear if a future redesign nests the mirror
  text inside another inline element with UA padding (`<kbd>`,
  `<samp>`, etc.). The safe pattern is to apply explicit `padding: 0;
  margin: 0;` to **whatever element** physically holds the mirror
  text, not just to the outer `<pre>`.
- The bundled artefacts under `dist/cogniflow-ui/_internal/.../static/`
  also need their copies of `style.css` / `app.css` refreshed when the
  PyInstaller bundle is next rebuilt; the source files are the
  authoritative versions.

### 19. Feature: `python app.py` auto-opens the browser to the homepage

**Request**
The bundled launcher (`launch.py`, used by the PyInstaller build) already
opens the user's default browser at `http://HOST:PORT` once uvicorn has
bound the socket. Running from source with `python app.py` did **not**
do this — the operator had to copy the URL out of the console and
paste it into a browser tab. The two launch paths should behave the
same so the dev/source experience matches the bundled-binary
experience.

**Implementation**
Mirrored the `_open_browser_when_ready` helper from `launch.py:26-39`
into the `if __name__ == "__main__":` block in `app.py`. It runs in a
daemon thread that sleeps `1.5 s` (so uvicorn has time to bind), then
calls `webbrowser.open(url, new=2)` and swallows any failure with a
log line — auto-open is a nicety, not a hard requirement.

The thread is started **before** `uvicorn.run("app:app", reload=True)`,
so the parent process arms the open, then hands control to uvicorn.
With `reload=True`, uvicorn spawns a worker subprocess that imports
`app` as a module (not as `__main__`), so the browser-open does not
fire on every reload — it fires exactly once, on initial startup.

`launch.py` was not touched: the bundled path keeps its own copy of
the helper. The two are intentionally a small duplicate (≈6 lines)
rather than a shared module — `launch.py` already imports from `app`,
and pulling the helper out into a third module just to avoid those
six lines isn't worth the import-graph noise.

**Files changed**
- `app.py` — `if __name__ == "__main__":` block now imports
  `threading, time, webbrowser`, defines a local
  `_open_browser_when_ready` helper, starts the daemon thread, then
  calls `uvicorn.run`.

**Smoke check performed**
Wrote a shim that monkey-patches `webbrowser.open` to print a sentinel
line, then `runpy.run_path("app.py", run_name="__main__")`. After
~6 s the shim's stdout contained:

```
INFO:     Uvicorn running on http://127.0.0.1:8000 (Press CTRL+C to quit)
INFO:     Application startup complete.
[BROWSER_OPEN_CALLED] url=http://127.0.0.1:8000
```

Confirms (a) the server binds before the browser fires, (b) the URL
matches `http://HOST:PORT` from `config.json`, (c) the open is called
exactly once.

**Forward-port note**
- `launch.py` still has its own `_open_browser_when_ready`. If a v2
  refactors that helper (longer delay, retry-on-failure, port probe
  instead of fixed sleep), apply the same change to `app.py`'s copy
  or extract a shared `_browser.py` module.
- The 1.5 s delay is a heuristic. On slow machines uvicorn may not
  have bound the socket yet — the browser then loads a "connection
  refused" page and the user has to refresh. A more robust approach
  is to poll the port (`socket.create_connection(...)`) until it
  accepts, with a timeout cap, and only then fire the open. Left as
  a follow-up because the current heuristic matches the bundled
  launcher's behaviour and the user has not reported the race.
- If the user wants `uvicorn app:app --reload` (direct CLI) to also
  auto-open, that path doesn't enter `__main__`, so the helper isn't
  invoked. Either tell operators to use `python app.py` or add a
  startup hook on the FastAPI app — but a startup hook fires on
  every reload-restart, so it would need a "first-run" guard via
  an env var (e.g. `if os.environ.get("UVICORN_RELOAD_RUN") != "1":`)
  to avoid spawning a tab per code change.

### Re-deployment note for the bundled binary

The fixes above touch Python source (`app.py`) and Jinja2 templates. Templates
under `dist/cogniflow-ui/_internal/configurator/templates/` are read at
runtime and need to be updated alongside the source if the bundled binary in
`dist/` is redistributed. The Python changes require a PyInstaller rebuild
(`cogniflow-ui.spec`) for the bundled binary; running from source picks them
up on server restart.
